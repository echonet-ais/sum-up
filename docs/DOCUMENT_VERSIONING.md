## 1. 개요

### 1.1 목적
본 문서는 SumUp 프로젝트의 문서 버전 관리 방법을 정의합니다. 문서의 변경 이력을 추적하고, 문서 간 일관성을 유지하기 위한 가이드라인을 제공합니다.

### 1.2 범위
- 문서 버전 관리 규칙
- 변경 이력 기록 방법
- 문서 상태 관리
- 문서 간 일관성 유지

---

## 2. 버전 관리 규칙

### 2.1 버전 형식
문서 버전은 **시맨틱 버저닝(Semantic Versioning)**을 따릅니다:

```
MAJOR.MINOR.PATCH
```

- **MAJOR**: 주요 변경사항 (호환되지 않는 변경)
- **MINOR**: 기능 추가 (하위 호환성 유지)
- **PATCH**: 버그 수정 (하위 호환성 유지)

### 2.2 버전 증가 규칙

#### MAJOR 버전 증가
- 문서 구조의 큰 변경
- 문서 형식 변경
- 호환되지 않는 내용 변경

#### MINOR 버전 증가
- 새로운 섹션 추가
- 기능 설명 추가
- 예시 코드 추가

#### PATCH 버전 증가
- 오타 수정
- 문장 개선
- 링크 수정
- 형식 오류 수정

## 3. 문서 헤더 형식

### 3.1 표준 헤더
모든 문서는 다음 형식의 헤더를 포함해야 합니다:

```markdown
> **프로젝트**: SumUp  
> **버전**: 1.0.0  
> **날짜**: 2025-11-29  
> **상태**: 초안 | 진행중 | 완료
```

### 3.2 상태 값
- **초안**: 작성 중이거나 검토 전
- **진행중**: 검토 중이거나 수정 중
- **완료**: 최종 승인 완료

---

## 4. 변경 이력 기록

### 4.1 변경 이력 테이블
모든 문서는 문서 하단에 변경 이력 테이블을 포함해야 합니다:

```markdown
## 문서 이력

| 버전 | 날짜 | 작성자 | 변경사항 |
|------|------|--------|---------|
| 1.0.0 | 2025-11-29 | Initial | 초기 문서 작성 |
| 1.1.0 | 2025-11-29 | John Doe | API 문서 섹션 추가 |
| 1.1.1 | 2025-12-01 | Jane Smith | 오타 수정 |
```

### 4.2 변경사항 작성 규칙
- **명확성**: 변경사항을 명확하게 작성
- **간결성**: 핵심 내용만 포함
- **일관성**: 동일한 형식 유지

### 4.3 변경사항 예시
```
✅ 좋은 예:
- API 엔드포인트 섹션 추가
- 배포 가이드 Vercel 설정 추가
- 오타 수정 (API → API Client)

❌ 나쁜 예:
- 수정
- 업데이트
- 변경
```

## 5. 문서 업데이트 프로세스

### 5.1 문서 수정 전
1. **이슈 생성**: 문서 수정 이유와 범위 명확화
2. **현재 버전 확인**: 문서의 현재 버전 확인
3. **영향 범위 분석**: 다른 문서에 미치는 영향 확인

### 5.2 문서 수정 중
1. **버전 증가**: 변경 규칙에 따라 버전 증가
2. **날짜 업데이트**: 수정 날짜 업데이트
3. **상태 업데이트**: 필요시 상태 변경
4. **변경 이력 추가**: 변경 이력 테이블에 항목 추가

### 5.3 문서 수정 후
1. **검토**: 내용 검토 및 오타 확인
2. **링크 확인**: 내부/외부 링크 확인
3. **커밋**: 의미 있는 커밋 메시지 작성
4. **문서 인덱스 업데이트**: `docs/README.md` 업데이트 (필요시)

## 6. 문서 간 일관성

### 6.1 프로젝트 이름
- 모든 문서에서 "SumUp"으로 통일
- 저장소 URL: `https://github.com/gr22nist/sum-up`

### 6.2 날짜 형식
- 날짜 형식: `YYYY-MM-DD` (예: 2025-11-29)
- 시간이 필요한 경우: `YYYY-MM-DD HH:mm` (예: 2025-11-29 14:30)

### 6.3 용어 통일
- 일관된 용어 사용
- 약어 사용 시 첫 사용 시 풀네임 표기

### 6.4 링크 형식
- 내부 링크: 상대 경로 사용 (예: `[PRD.md](./PRD.md)`)
- 외부 링크: 전체 URL 사용 (예: `[Next.js](https://nextjs.org)`)

## 12. 자동화 (선택)

### 12.1 문서 버전 검증
문서 헤더에 버전이 포함되어 있는지 확인하는 스크립트:

```bash
#!/bin/bash
# scripts/check-doc-version.sh

for file in docs/*.md; do
  if ! grep -q "버전:" "$file"; then
    echo "Warning: $file does not have version header"
  fi
done
```

### 12.2 변경 이력 검증
변경 이력 테이블이 있는지 확인하는 스크립트:

```bash
#!/bin/bash
# scripts/check-doc-history.sh

for file in docs/*.md; do
  if ! grep -q "문서 이력" "$file"; then
    echo "Warning: $file does not have change history"
  fi
done
```
